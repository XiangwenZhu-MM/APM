

# 智汇资产平台 （Smart Asset Hub ）

# 产品愿景

我们的目标是打造一个以“设备管理”为核心、AI驱动、开放协同的工业软件平台。我们将APM定位为应用层的核心底座，补齐 WAGO SCADA 上层应用的功能短板，实现资产主数据的统一沉淀与治理。通过深度集成最新的AI技术，我们不仅提供传统APM的资产台账与全生命周期管理，更赋予平台智能预测与辅助决策的能力，这也是我们区别于传统产品的核心竞争力。该平台将极大地赋能上层应用开发，加速行业项目落地，降低客户的业务创新成本，从而帮助我们覆盖更多资产密集型客户，成为企业数字化转型的坚实底座。

# 原因和背景

开发APM是基于下面的现实和战略考量：

- 我们已经拥有成熟的 WAGO SCADA 产品，并以其作为数据通信和可视化底座，形成了“以 WAGO SCADA 为基础、上层开发 App”的平台模式。例如，现有的 EMS 产品就是基于 WAGO SCADA 之上开发的典型应用。
- 但 WAGO SCADA 本身并不具备设备管理功能，而上层的许多 App（如 EMS）又各自实现了设备管理，导致设备主数据分散、重复开发，难以实现全局统一和高效协同。
- 因此，我们希望将设备管理能力，以及设备数据与设备主数据的关联能力，统一沉淀在 APM 平台中。这样，上层 App 可以专注于设备数据的分析和业务创新，无需重复开发设备管理模块。
- 我们已经在滇池水务，格德格瑞项目上积累一些设备管理的经验和可复用的代码
- 未来，公司设备集成组的 iDM 平台也可以作为类似 WAGO SCADA 的数据源接入 APM，进一步丰富数据来源和集成能力。
- 对于未来有 IoT 平台需求的客户，也可以通过 APM 平台，将其网关设备作为数据源直接接入，实现灵活扩展和统一管理。

这样，APM 不仅补齐了 WAGO SCADA-EMS 体系中的设备管理短板，还为未来多源数据集成、统一资产主数据和上层业务创新提供了坚实的数字底座。

目前我们在国内销售 WAGO SCADA 源码，部分客户（如信卓、格德格瑞）还需要上层的设备管理与数据分析能力。通过提供并整合 APM 与 WAGO SCADA，客户可在平台上快速构建应用，显著降低业务软件开发成本、加速创新落地，同时形成产品协同的“1+1>2”效应，带动整体销售增长。

# 技术架构 (LZG)

本章节仅描述**技术架构要求与应达到的效果**，不绑定具体产品/中间件/框架选型；实现时可在满足约束的前提下替换组件与落地方式。

## 1. 架构目标（需要达到的效果）

1. **统一界面入口（一个系统体验）**：用户视角下，所有功能模块应表现为“一个系统”，实现统一入口、统一导航、统一身份、统一主题/语言与一致交互反馈；模块间切换应尽量无割裂。
2. **统一资产主数据**：所有上层应用（EMS、未来 OEE/环保/维保等）对“设备/位置/关系/测点绑定/资产属性”使用同一套权威数据源与 API，避免重复建模与数据孤岛。
3. **动静结合**：静态主数据（台账、结构、属性）与动态数据（测点、事件、时序指标、工单/告警等）可在同一业务上下文中被关联、查询与分析，并可按资产/位置回溯。
4. **配置/插件驱动扩展**：新行业能力通过“启用插件 + 配置”上线，尽量减少定制开发；插件具备可复用、可部署、可隔离与可治理特性（启用/停用/升级/回滚）。
5. **多维指标体系**：支持按任意业务维度（楼层/产线/设备类型/状态/自定义标签等）进行过滤、分组、下钻、趋势对比；维度扩展可配置且口径一致。
6. **国产化与内网部署友好**：支持纯内网、离线/弱网、国产软硬件环境；外部依赖可替换且不形成强绑定。
7. **可观测、可运维、可演进**：链路追踪、审计、备份恢复、滚动升级、数据保留与归档策略具备可操作性；支持分阶段演进与平滑迁移。

## 2. 分层与模块边界（逻辑架构要求）

平台逻辑上应至少拆分为以下层次，各层通过明确契约解耦；禁止跨层直接耦合底层存储或内部实现细节。

### 2.1 统一界面入口（统一工作台/门户）

- 作为唯一用户入口，负责全局导航、路由、身份、权限、主题、多语言、消息中心与全局交互规范治理；自身不承载具体行业业务逻辑。
- 支持模块化集成与独立交付，各业务模块可独立演进、独立部署；入口在运行时按配置启用/禁用与装配。
- 页面与导航支持配置驱动，可在不整体发布的前提下调整信息架构与模块可见性（受权限控制）。
- 提供跨模块一致的交互体验（加载/错误/成功提示、表单规范、通知与待办呈现等），降低学习成本。
- 对关键协同能力（模块清单、导航树、页面编排、用户与权限、主题与多语言等）定义稳定契约，并具备版本管理与兼容策略。

### 2.2 统一数据接入层（Adapters）

- 支持从 WAGO SCADA、iDM、IoT 网关以及第三方系统（ERP/MES/CMMS 等）接入数据。
- 接入层负责规范化事件结构、鉴权、限流、幂等与基础校验，形成可复用的接入治理能力。

### 2.3 事件/消息骨干（Event Backbone）

- 解耦生产者与消费者；支持至少一次投递；支持回放（replay）与按时间范围重放。
- 需支持对事件的全链路追踪（来源、处理状态、失败原因、重试/回放记录），并可用于问题定位与运维操作。
- 事件需包含租户标识、幂等键、业务时间、接入时间、来源标识、事件类型/版本等元信息。

### 2.4 规则与指标计算层（Rules & KPI Engine）

- 支持声明式规则/指标定义（来自行业插件配置），并具备运行时解释执行能力。
- 至少覆盖两类典型计算：
  - 采样型累计量 → 支持差分/增量计算与状态管理（含故障恢复）
  - 事件型计数 → 支持窗口聚合与“空窗口补零”
- 支持乱序/迟到数据处理策略（可配置容忍窗口/水位线等）。
- 支持回补（backfill）与重算（recompute），确保规则变更后可再生历史指标并可追溯依据。

### 2.5 维度打标/语义关联层（Tagging & Context Enrichment）

- 从统一资产模型提取维度（位置层级、资产类型、组织、标签、上下游关系等），注入到派生指标的维度标签中。
- 高频读取需支持缓存与失效策略；维度变更后应可触发增量更新/重打标。

### 2.6 存储层（按职责分库/分域，不限定实现）

- 关系型主数据存储（强一致）：资产类型、资产实例、子模型、测点元数据、插件注册信息、权限与审计等。
- 时序/指标存储（高写入高压缩）：统一指标模型，支持按时间与标签检索、聚合、降采样与保留策略。
- （可选）关系/拓扑增强：用于多跳关系、影响分析、根因路径、血缘追踪等；不用于大规模时序聚合。
- （可选）离线分析/报表增强：用于大规模离线统计、宽表分析与高并发报表。

### 2.7 统一查询与开放接口层（API）

- 对外提供统一的资产 API、指标查询 API、规则/插件管理 API，以及可选的标准兼容接口。
- API 版本化（v1/v2 等）、接口文档与契约测试机制可落地；支持面向服务间调用的统一鉴权与审计。

### 2.8工作流引擎

- 支持行业特定的业务流程模板，包含：流程步骤、角色权限、审批规则、触发条件
- 支持流程的可视化配置与版本管理

### 2.9 多租户与数据隔离

- 多租户实现模型（物理隔离 vs 逻辑隔离）
- 安全边界：跨租户访问审计、租户管理员权限模型、合规隔离（例如为监管客户提供物理隔离）

### 2.10 边缘/离线/弱网支持

- 双向同步策略（Delta sync），冲突解决规则，能在边缘执行规则、做本地缓存、离线告警并在连通时回传

### 2.11 模型/ML 能力

- 模型生命周期管理（Model Registry）：模型版本、训练数据快照、评估指标、审批流与回滚。
- 可解释性与合规：决策链审计，方便运维/合规排查。

## 3. 统一资产模型（主数据）能力要求

平台需支持：

1. **固定核心结构 + 扩展字段机制**
    - 平台需要面向不同行业与设备类型（例如电表、粮机、以及未来新增行业设备）进行资产建模。这些资产既存在一部分**共享字段**（如编码、名称、位置、状态、归属等），也存在大量**类型特有字段**（如不同设备需追踪的技术参数、铭牌信息、运行约束、验收口径等）。因此需要在**不改变存储机制与接口语义稳定性**的前提下，实现“同一套资产体系可持续扩展字段”的能力。
    - **例如**：
        - 资产类型模板（Asset Type）用于定义该类型资产的字段元数据（schema），包括字段标识、名称、数据类型、单位/枚举、必填/默认值、校验规则、是否可检索/可聚合、展示分组等。
        - 资产实例（Asset Instance）存放固定核心字段与扩展字段值（properties），并可依据“资产类型模板”进行字段校验与一致性约束。
    - 扩展字段机制应避免通过频繁结构变更来适配新类型字段，从而降低升级与运维风险；同时应支持按需建立索引/检索能力，确保在常用查询场景下具备可控的性能与成本。

2. **子模型（Submodel）挂载**
	- 同一资产可挂载多个子模型（维保、能效、健康、巡检等），子模型亦支持扩展属性与模板化。
	- 效果：核心资产稳定，业务能力按需叠加，降低模块耦合。

3. **测点绑定（Measurement Point）**
	- 将测点与资产建立绑定关系（测点标识 → asset_id + metric_name + unit + data_type 等）。
	- 支持批量导入、校验、继承/模板化绑定，以及（可选）插件提供绑定建议。

4. **多维关系表达**
	- 支持树形层级与多对多网状关系（供电关系、工艺关系、网络关系等）。
	- 对多归属/多跳复杂关系，需提供可选增强能力，避免关系查询复杂度失控。

## 4. 行业插件机制（平台级要求）

插件是“可复用、可部署、可隔离的行业解决方案包”，平台需要提供：

1. **插件包规范与注册**
	- 插件包含元数据（名称/版本/依赖/兼容性）、资产类型/子模型、KPI 定义、规则、看板/页面配置（以及可选扩展能力声明）。
	- 插件可启用/禁用、版本升级/回滚、依赖校验、冲突检测（同名 key、字段名冲突、指标冲突等）。

2. **声明式优先，扩展受控**
	- 大多数场景通过声明式配置落地；复杂逻辑允许可选扩展，但必须具备隔离与安全边界（资源限制、权限、审计）。

3. **运行时生效与最小化中断**
	- 插件加载后应自动注册资产类型、规则、看板菜单/页面配置等；不应要求整个平台长时间中断（如需中断需提供最小化窗口与回滚策略）。
	- 插件沙箱/资源配额：CPU/内存/磁盘/网络配额；插件运行在受控环境（容器/namespace）内以避免互相影响。

4. **可观测与可审计**
	- 插件引入的规则/指标计算需可追溯来源（插件版本、规则 ID、配置快照、计算版本等）。

---

## 5. 指标体系与时序数据（统一 Metrics）要求

1. **统一指标模型**
	- 指标写入遵循统一命名与标签规范，必须包含租户/项目边界信息。
	- 控制标签基数（禁止高基数随机维度直接作为标签），提供治理规则与告警/拦截机制。

2. **典型计算能力（必须）**
	- 累计差分：支持每 key 状态管理与故障恢复
	- 事件计数：窗口聚合 + 空窗口补零
	- 常用聚合：sum/avg/max/min/last/rate 等（按配置启用与组合）

3. **保留与归档**
	- 标签基数治理机制：禁止/告警高基数标签的产生；提供 tag mapping、tag cardinality limit、标签替代（如把随机维度做为字段而非标签）
	- 支持热/冷数据分层、降采样策略、保留周期可配置；历史回放与重算不依赖在线热数据永久保留。

---

## 6. 关键协同契约（前后端衔接点，必须标准化）

为实现“统一界面入口”的一致体验，至少需要标准化以下契约（均需版本化与向后兼容策略）：

- 用户与权限：用户信息、租户/项目边界、权限集/能力清单、审计要求
- 模块清单：模块启用状态、可见范围、入口/依赖信息（不绑定具体集成技术）
- 导航与页面编排：菜单树、页面定义、组件/区域编排配置、权限过滤结果
- 主题与多语言：主题变量/资源、语言包资源、按需加载与可覆盖策略
- 通知与待办（可选）：跨模块消息聚合、已读/未读与处理状态同步

---

## 7. 非功能性要求（SLA/SLO 可在项目阶段量化）

- 安全：租户隔离、最小权限、敏感信息保护、关键操作审计、可配置安全策略
- 可靠性：关键链路可降级与可恢复；一致性校验与异常补偿机制可落地
- 性能：高频写入与高并发查询可扩展；热点维度与常用查询具备优化空间
- 可运维：健康检查、告警、容量与成本可视化、可演练的备份恢复能力
- 可演进：数据模型、规则与接口支持版本化；提供迁移/兼容策略，避免升级“全量改造”


# 系统介绍

### 1.1 WAGO SCADA (Supervisory Control and Data Acquisition)

**WAGO SCADA** 是整个系统的**数据采集与监控底座**，作为**运营基础层**，负责与底层硬件（PLC、RTU 和传感器）进行实时交互。在产品生态中，它为 APM 提供实时的设备运行数据，是实现“动静结合”资产管理的前提。

* **实时数据采集：** 支持多种工业协议，高频率采集过程变量（如压力、温度、流量、电量等），为上层应用提供精准的实时数据流。
* **过程控制与逻辑：** 执行复杂的控制脚本和设定点管理，实现自动化的工业流程控制。
* **组态可视化 (HMI)：** 提供强大的组态能力，通过 HMI（人机界面）和动态图形，实现生产过程的直观监控与实时预警。
* **过程数据存储：** 支持多种高性能工业数据库（如时序数据库、关系型数据库等），实现海量业务数据的长周期存储。WAGO SCADA 通过标准化的接口屏蔽了底层的存储差异，使得 APM 及上层应用无需关心传感器数据是如何存储的，实现了业务逻辑与底层物理存储的彻底解耦。
* **定时任务与时间表 (Scheduler)：** 提供强大的时间表功能，允许上层应用根据业务需求配置定时任务。支持设备巡检、定时控制下发及跨系统数据同步等自动化操作，显著提升了系统的无人化作业能力。
* **工业报警功能：** 提供多维度的报警配置与实时推送机制，支持通过短信、微信、钉钉、手机 APP 及语音等多种渠道触达用户，并具备报警屏蔽、确认、统计及转发功能，确保生产中的关键异常能够第一时间得到响应与闭环处理。
* **AI 智能组态与业务协同：** 集成 WAGO SCADA AI 助手，支持通过自然语言描述自动生成监控看板和业务界面（Text-to-Dashboard），实现秒级 UI 渲染。AI 助手可自动解析第三方 RESTful API，实现 WAGO SCADA 与 ERP、MES 等系统的数据融合与业务联动，支持在组态界面直接触发外部系统操作。该能力使 WAGO SCADA 从单一监控平台跃升为工业全业务智能协同门户。


### 1.2 智汇资产平台 (Smart Asset Hub)

**Smart Asset Hub** 是整个生态系统的“管理底座”与“智能大脑”。它通过将传统的资产管理流程（EAM）与基于实时数据的性能分析（APM）相结合，实现了资产全生命周期的闭环治理。

#### 核心管理底座 (Asset Foundation) — 规范业务流

此部分构建了资产的“数字户口本”，解决资产信息碎片化与流程不标准的问题。

1. **资产全生命周期管理 (Asset Management)**
* **资产台账与层级结构**：支持建立“父子关系”树，实现从系统到组件的精细化管理（如：中央空调系统  压缩机  控制板）。
* **位置与状态追踪**：定义资产的物理或逻辑位置，并实时记录运行读数（如累计运行小时、里程等），作为自动化维护的基准。
* **标准化故障体系**：建立“类-问题-原因-措施”的 Failure Codes 体系，为后续 AI 故障挖掘提供标准化标注。


2. **精益工单引擎 (Work Management)**
* **全闭环管控**：覆盖从作业申请、审批、派工到执行汇报、归档的全流程。
* **标准作业计划 (SOP)**：预设作业计划（Job Plans），明确维修所需的工时、工具与物料，确保维护质量一致性。
* **策略驱动维护 (PM)**：支持基于时间（定期）或基于读数（如满 1000 小时）自动生成预防性维护工单。


3. **资源与备件管理 (Inventory Management)**
* **物资供应协同**：关联资产与物料主数据，管理仓库货位，确保维修现场“有件可用”。
* **自动补货逻辑**：通过安全库存设定，在备件不足时自动触发采购申请（PR）。


---

#### 高级智能引擎 (Asset Intelligence) — 释放数据价值

此部分深度集成 WAGO SCADA 的实时数据，通过 AI 技术实现从“被动维修”到“主动预测”的跨越。

* **实时感知与异常监测 (Monitor)**：
- 通过 WAGO SCADA 实时接入压力、温度、振动等传感器数据。
- 利用 AI 异常检测算法捕捉微小偏差，在故障成型前发出早期预警。


* **资产健康画像 (Health)**：
- 结合实时工况、历史维修记录及巡检结果，为每台核心资产生成 0-100 的实时健康评分。
- 评估资产剩余使用寿命（RUL），帮助管理层决定是“继续维护”还是“投资更换”。


* **预测性维护决策 (Predict)**：
- 基于机器学习模型预测特定故障（如轴承过热、电机短路）发生的概率及故障时间轴。
- AI 助手自动解析 API，实现跨系统业务联动，将预测结果直接转化为维修指令。

### 核心价值：智汇协同效应

* **智能进化**：通过 AI 智能组态，实现从“人工配置”到“智能生成”的跨越，大幅提升了开发与运维效率。

### 1.3 EMS (Energy Management System)

**EMS** 是专注于资源效率的**专业优化工具**。

* **能耗监测：** 监控全厂的能源使用（电、水、气、蒸汽）。
* **绩效基准：** 计算能耗强度和碳足迹。
* **异常检测：** 识别能源浪费或低效运行模式。
* **ISO 50001 支持：** 深度集成 ISO 50001 能源管理体系标准，内置 PDCA 能源管理循环，助力企业实现体系化、持续改进的能效管理。

* **合规与审计：** 提供自动报告引擎，支持一键导出符合 ISO 50001 审计要求的能源管理评审报告，便于合规性证明。
* **节能项目追踪：** 记录和量化节能改造项目的成效，实现节能量的可量化管理。

* **国际标准兼容：**
	* **ISO 50001：** 能源管理体系国际标准，指导企业建立PDCA循环的能效管理流程，实现持续改进和合规运营。

# AI能力与业务协同 (LZG)

平台集成了以**大语言模型（LLM）为核心的 AI 助手能力**（如 WAGO SCADA AI 智能组态），并与统一资产模型、指标体系和业务流程深度融合，使 AI 成为工业软件平台中的**通用智能协同层**，而非孤立的辅助功能。

通过自然语言、多模态识别与配置化生成能力，平台实现了从“人工建模与配置”向“AI 理解业务并生成系统能力”的跃迁，全面提升系统易用性、配置效率与业务协同水平。

## 1. 自然语言驱动的界面与配置生成（Text-to-Dashboard）

平台支持通过自然语言直接生成系统界面与业务配置：

* 用户可使用自然语言描述业务需求（如监控对象、指标、对比维度、展示方式等）

该能力使业务人员无需掌握复杂组态或开发技能，即可快速构建符合业务语义的监控看板和业务界面，实现秒级 UI 渲染与数据闭环。

AI 不仅理解“文字描述”，还可将传统工业场景中大量存在的**非结构化或半结构化载体**转化为系统可执行的业务配置。

针对物流、仓储、采购等仍以纸质单据为主的业务场景：

* 支持扫描或拍照上传出入库单、物流交接单、随货单等
* AI 自动识别关键字段（物料、数量、时间、责任人等）

平台将 AI 作为统一的语义查询入口，使业务人员可以“直接问系统”。

## 4. AI 原生平台：能力可发现、可组合、可治理（AI-Friendly Platform）

本节强调的不是“AI 能做什么功能”，而是 **平台如何以工程化方式让 AI 可用、可控、可上线**：AI 能理解平台有哪些能力与规则，并在约束内自动组合，帮助普通用户用自然语言快速落地业务需求，而无需阅读/掌握传统插件开发的契约细节。

### 4.1 能力目录与语义契约（Capability Catalog）

平台应将关键能力以“机器可读 + 语义可解释”的方式登记为能力目录，供 AI 检索与推理编排：

- **资产与主数据能力**：资产类型/字段 schema、子模型、测点绑定、关系模型、校验规则、导入/对账能力
- **指标与规则能力**：指标口径、窗口/聚合方式、乱序/迟到处理、回补重算策略、状态管理约束
- **页面与交互能力**：页面定义 schema/DSL、组件库能力、筛选/下钻/联动规范、导航与权限过滤规则
- **工作流能力**：流程模板、步骤与角色、审批规则、触发条件、版本与回滚
- **集成与事件能力**：适配器类型、鉴权方式、限流/幂等要求、事件语义（type/version）、重试与死信策略

能力目录至少应包含：版本号、输入/输出 schema、约束条件、示例（few-shot）、错误码语义、以及租户/权限边界说明。

### 4.2 契约优先的生成与编排（Contract-first Orchestration）

AI 的输出必须落在平台的**标准契约**上（可审阅、可校验、可回滚），避免生成“不可治理”的脚本/私有逻辑：

- 生成页面：输出符合页面编排 schema 的配置（组件/布局/交互/权限）
- 生成规则与指标：输出符合规则 DSL 的声明式定义（含口径说明与版本）
- 生成流程：输出流程模板与审批规则配置（含角色映射与审计点）
- 生成集成：输出连接器配置与字段映射（含幂等键、重试策略、事件类型版本）

平台应提供：Schema 校验 + 业务校验（字段合法性、指标命名规范、标签基数风险、跨租户风险等），在发布前给出可操作的诊断与修复建议。

### 4.3 护栏与治理（Guardrails）

为确保 AI 编排“可控可用”，平台需提供明确护栏：

- **权限护栏**：AI 只能在用户授权范围内读取能力目录与数据；启用插件/发布规则/回补重算/外部写入等关键动作需二次确认或审批
- **资源护栏**：对回补、批量导入、全量重算等高成本操作提供配额、限流与影响评估（预计计算量/影响范围/回滚点）
- **数据护栏**：租户隔离、敏感字段脱敏、访问审计；生成内容默认遵循最小权限与合规策略

### 4.4 可观测、可追溯、可回滚（Provenance）

所有由 AI 生成/修改的对象（页面、规则、流程、映射等）必须具备可追溯元信息与回滚能力：

- 来源：用户/会话、引用的能力目录版本、配置快照、（可选）模型版本
- 变更：diff、审批记录、发布人、发布时间、回滚点
- 运行：规则/指标计算版本、失败原因、重试/回放记录与效果评估（可选）

## 5. AI 驱动的跨系统数据融合与业务联动

AI 助手可自动解析并理解第三方系统的接口语义：

* 支持对接 ERP、MES、CMMS 等外部系统
* 在统一界面中展示多源数据，消除系统割裂
* 支持在组态界面或业务页面中，直接触发外部系统操作

通过 AI 参与系统集成与语义映射，跨系统联动从“定制开发”转变为“配置驱动”。

## 6. 价值与交付方式：从“懂平台的人”到“懂业务的人”也能构建应用

平台带来的核心变化是：把“平台规则与能力组合”的门槛，从开发者/专家侧转移给 AI 与平台治理机制承担。

- **面向业务用户**：用自然语言表达目标 → AI 生成可执行配置 → 人工确认/审批 → 上线运行（全程可追溯）
- **面向 ISV/集成商**：减少从零阅读契约与重复搭建的成本，把精力集中在行业口径与差异化能力；交付物以“可版本化配置与插件包”为主，便于升级与复用
- **面向运维与管理者**：关键变更可审计、可回滚；跨系统联动与规则计算具备可观测与故障定位路径

## 7. 小结

通过平台级“能力可发现/可编排/可治理”的设计，AI 在平台中不只是对话入口，而是能在明确契约与护栏内，把用户意图转换为可运行的页面、规则、流程与集成配置，实现：

* 从 **读文档/懂契约** 到 **自然语言 + AI 编排 + 人工确认**
* 从 **单点功能生成** 到 **跨资产/指标/流程/集成的组合式落地**
* 从 **可演示** 到 **可上线、可审计、可回滚的工程化 AI 能力**

# 未来可能性（LZG）

本平台在设计之初即以“统一框架 + 行业插件 + 可演进能力”为核心原则，目标不是一次性覆盖所有业务场景，而是构建一个**能够随着业务复杂度、行业深度和智能化水平持续演进的资产与运维数字底座**。以下从总体思路与具体扩展方向两层，系统性说明平台的未来可能性。


## 一、总体扩展思路

所有未来扩展均遵循同一技术范式：

* **统一资产与指标框架**：所有新行业、新业务能力都以统一资产模型（Asset Model）、指标体系（Metric/KPI）和事件语义为基础，避免形成新的“烟囱系统”。
* **插件化与可组合能力**：通过标准化插件机制，将行业能力、算法模型、业务规则、UI 模块作为独立插件接入平台，实现按需启用、独立升级与快速试错。
* **事件驱动与可回溯架构**：所有核心业务变化（状态、告警、工单、指标变更）均通过事件流进行传播，支持回放、重算和模型再训练，为智能化和合规审计提供基础。
* **数据、模型与流程解耦**：数据采集、规则计算、AI 模型、业务流程相互解耦，使平台可以在不破坏既有能力的情况下不断引入新算法、新流程和新视图。

基于以上原则，平台不仅能够覆盖当前 APM/运维管理需求，也能够自然生长出面向不同垂直行业和高级智能场景的能力。

---

## 二、面向垂直行业与业务场景的扩展方向

### 1. 新能源场站（光伏 / 储能）
- **场景示例**：自动识别组串失配并生成运维任务  
  - 在资产模型中定义“光伏组串”类型，关联“电流”“电压”测点；  
  - 通过数据接入框架订阅 SCADA 的实时遥测；  
  - 配置规则：“若同一逆变器下任意两组串电流差值 > 20% 且持续 10 分钟 → 标记为‘疑似失配’”；  
  - 绑定工作流：自动创建“现场核查”工单，并推送至运维 App。  
- **平台能力复用**：资产建模 + 动态数据 + 规则引擎 + 工单工作流。

### 2. 数据中心（IDC）
- **场景示例**：冷水机组能效异常自动诊断  
  - 定义“冷水机组”资产类型，挂载“COP”“冷冻水进出温差”“运行负载率”等计算字段；  
  - 平台自动从 BAS 系统采集原始数据，并按预设公式计算 COP；  
  - 配置规则：“若 COP < 4.0 且负载率 > 70% → 触发‘能效偏低’事件”；  
  - 在可配置看板中展示异常设备列表，点击跳转至设备详情页（由 UI 框架自动生成）。  
- **平台能力复用**：资产模型 + 数据计算表达式 + 规则引擎 + 元数据驱动 UI。

### 3. 智能工厂
- **场景示例**：产线设备 OEE 自动计算与停机归因  
  - 在资产树中建立“产线 → 工位 → 设备”结构；  
  - 接入 PLC 的“运行/停机/报警”状态信号；  
  - 配置 OEE 计算规则（可用平台内置模板），并定义停机原因分类（如“换模”“缺料”）；  
  - 当设备停机超阈值，自动弹出移动端表单供操作员选择原因，数据回写至资产实例。  
- **平台能力复用**：层级资产 + 状态数据接入 + 规则触发 + 移动端表单配置。

### 4. 智能楼宇
- **场景示例**：AHU 过滤器压差超限自动派单  
  - 定义“AHU”资产，关联“初效/中效过滤器压差”测点；  
  - 配置规则：“压差 > 200Pa → 触发‘需更换过滤器’事件”；  
  - 绑定工作流：自动生成保养工单，指定责任人，并设置 SLA 倒计时；  
  - 工单状态变更后，自动更新资产的“上次保养时间”属性。  
- **平台能力复用**：资产属性管理 + 规则引擎 + 工作流 + 属性联动。

### 5. 厂务系统（半导体 / 生物医药）
- **场景示例**：纯水系统电导率超标联动 SOP  
  - 建立“纯水分配单元”资产，接入电导率、流量、压力等实时数据；  
  - 配置合规规则：“电导率 > 1.0 μS/cm → 触发‘水质异常’事件”；  
  - 绑定工作流：自动加载预置的 GMP 应急 SOP 步骤（如“停止供水 → 启动冲洗 → 通知 QA”）；  
  - 每步操作需在 App 中确认，形成完整审计追踪链。  
- **平台能力复用**：高精度数据接入 + 合规规则库 + SOP 工作流 + 审计日志。

通过统一的技术框架和清晰的扩展机制，平台的未来可能性并非零散功能叠加，而是一条清晰的演进路径：

> **从资产监控 → 运维管理 → 行业能力 → 智能决策 → 数字孪生与知识中枢**。

这种设计确保平台能够在保持架构稳定性的同时，持续吸收新的行业需求与技术能力，避免重构式升级，为长期发展提供坚实基础。

# 市场定位

## 产品定位
Smart Asset Hub 是专为 WAGO SCADA 客户设计的**智能资产管理开发平台**，提供标准化的设备管理功能模块与开放的可扩展架构，帮助客户快速构建上层业务应用。

## 目标客户群体

### 第一类：有设备管理需求的 WAGO SCADA 客户
这类客户已经部署了 WAGO SCADA 用于数据采集与监控，但在设备管理方面存在明显短板。他们通常是制造业、水务环保、能源电力等行业的生产企业，设备数量多、运维复杂度高。

他们的典型需求包括：
- 建立统一的设备电子台账，替代手工或 Excel 管理
- 实现预防性维护计划，减少突发故障停机
- 跟踪设备维修历史，分析故障规律
- 管理备件库存，关联维修工单

目前他们要么自行开发简单模块（功能有限、难以维护），要么考虑引入第三方 EAM 系统（集成复杂、成本高昂）。Smart Asset Hub 为他们提供了**中间路线**：在已有 SCADA 基础上直接扩展设备管理能力，无需推翻重来或复杂对接。

### 第二类：基于 SCADA 开发上层应用的客户
这类客户主要包括自动化集成商、行业软件开发商（ISV）以及企业内部的 IT 开发团队。他们不仅需要设备管理功能，更需要在 WAGO SCADA 之上构建完整的行业解决方案，如能源管理系统、生产执行系统、智能巡检平台等。

他们的核心诉求是：
- 获得稳定、标准的设备数据模型和 API 接口
- 复用成熟的设备管理功能，避免重复开发
- 集成先进的 AI 能力（如预测性维护、智能诊断）
- 支持快速定制和扩展行业特有功能

对于他们来说，Smart Asset Hub 不仅是一个功能模块，更是一个**开发底座**，能显著降低开发门槛、缩短交付周期、提升解决方案的竞争力。

## 市场竞争优势

相比客户自行开发，Smart Asset Hub 提供了成熟、标准的功能模块，节省了大量开发与测试时间。相比采购第三方独立系统，我们与 WAGO SCADA 的原生兼容性消除了复杂的集成成本与数据壁垒。我们的平台化架构和内置 AI 能力，使得客户不仅能解决当下的管理需求，更能构建面向未来的智能运维体系。

此外，随着未来更多行业APP（如EMS等）的不断集成与发展，Smart Asset Hub 将与这些应用形成强大的协同合力，进一步拓展平台的行业覆盖面和用户基础。通过平台化架构和开放接口，行业APP能够无缝对接、数据互通，形成“1+1>2”的生态效应，助力企业实现全方位的数字化转型，持续提升平台的市场竞争力和客户粘性。


# 🗺️ 内部研发导向 Roadmap  
> *注：各阶段以可运行 MVP 为目标，按人员空闲与项目机会灵活推进*

---

### 🔹 阶段一：统一门户与系统集成（MVP）

#### ✅ 核心目标

构建一个**可实际运行的统一门户入口**，作为各业务子系统的统一承载与治理框架，实现现有 SCADA 平台与设备管理系统的集成运行，并为后续能力演进（统一资产、AI、流程协同）奠定基础。

---

#### 📦 功能范围

##### 1. **统一前端门户与子系统治理**

* 提供统一的前端门户框架，作为所有子系统的唯一访问入口
* 支持在系统设置中：

  * 启用 / 禁用接入的子系统
  * 加载子系统暴露的功能模块与页面
  * 对子系统提供的菜单项进行**统一编排、重命名与目录树重排**
* 门户负责统一的：

  * 登录入口与统一认证机制
  * 用户、角色与基础权限框架（可对齐并复用 SCADA 既有站点权限模型）
  * 页面布局、导航风格与基础交互规范

> 统一门户本身不承载具体业务逻辑，而是对子系统进行**装配、治理与体验统一**。

---

##### 2. **系统集成与运行态协同**

* 通过 iframe 或微前端方式集成 WAGO SCADA 页面
* 支持在统一门户中直接访问：
  * SCADA 监控画面与基础操作界面
  * SCADA 实时测点数据
  * 设备管理系统界面 
* 支持统一认证与权限透传，避免重复登录

---

##### 3. **设备 – SCADA 测点关系维护与同步**
* 在设备管理系统中创建或变更设备参数时，可同步更新 SCADA 中对应的结构或测点定义
* SCADA 实际采集的数据、报警状态可基于映射关系回溯到设备维度进行展示
* 本阶段以**结构与映射关系打通**为目标，不引入复杂规则或指标计算逻辑

---

##### 4. **可选展示能力：AI 界面生成（演示级）**

* 集成已有 AI 能力（如：“显示 1# 水泵的电流和温度趋势”）
* AI 基于已建立的设备与测点映射关系：

  * 自动生成临时监控看板页面
  * 支持基础趋势展示与实时数据查看
* 该能力：

  * 仅作为 AI 能力展示与验证
  * 不作为正式生产配置路径
  * 不影响核心功能交付

---

#### 🧪 交付物

* 可部署的统一门户 Web 应用
* 子系统启用 / 禁用与导航配置管理能力
* SCADA 与设备管理系统的集成运行实例
* 设备 – 测点映射管理模块（含导入 / 导出）
* （可选）AI 看板生成演示 Demo


---

### 🔹 阶段二：统一资产模型与轻量规则引擎

#### ✅ 核心目标  
在统一门户基础上，引入**可扩展的资产模型**作为新数据底座，并实现最小化的事件处理与派生指标能力。

#### 📦 功能范围
1. **统一资产模型服务**
   - 支持自定义资产类型（Schema），包含属性、关系、层级结构
   - 资产实例管理（增删改查、状态管理）
   - 提供 RESTful API 供前端及未来插件调用
   - 支持从旧设备管理系统迁移数据（保留原 ID 映射）

2. **轻量事件与规则引擎**
   - 接收 SCADA 测点变化、告警等事件（通过消息队列或轮询）
   - 支持简单规则配置：`IF {tag} > {threshold} THEN raise alert`
   - 支持派生指标计算（如日运行小时、平均功率）
   - 规则执行结果可写入资产状态或触发通知

3. **增强可视化**
   - 基于新资产模型的资产树浏览
   - 支持按资产查看关联测点、告警、派生指标

4. **保留 AI 界面生成能力**
   - 使其能识别新资产模型中的设备与指标

#### 🧪 交付物
- 资产模型管理后台（含 Schema 编辑器）
- 事件监听 + 规则引擎核心模块（可配置、可重启）
- 新版资产中心页面（替代旧设备管理视图）
- 派生指标 API 与示例看板

> ⚠️ 此阶段**暂不实现**：插件市场、多版本管理、复杂流处理、多租户隔离。

---

### 🔹 阶段三：平台开放化与智能深化

#### ✅ 核心目标  
将平台能力产品化，支持行业扩展，并引入 AI 辅助决策与跨系统协同。

#### 📦 功能范围
1. **插件化架构**
   - 定义插件开发规范（资产扩展、KPI、页面、API）
   - 支持插件动态加载与配置
   - 内置 1–2 个行业插件（如 EMS 能效、BMS 楼宇）

2. **AI 原生能力**
   - 模型注册与管理（支持预测性维护等场景）
   - 智能工单推荐、故障根因分析
   - 自然语言问答（NLQ）对接统一指标库

3. **跨系统协同**
   - 与 ERP/MES 等系统通过标准接口联动
   - 统一工作流可触发外部操作（如创建采购申请）

4. **其它能力**
   - 移动端 App（巡检、告警处理）
   - 离线边缘同步（Delta 同步 + 冲突解决）
   - 碳核算、数字孪生可视化等垂直场景支持

#### 🧪 交付物
- 插件 SDK 与示例
- AI 服务模块（可插拔）
- 跨系统集成适配器框架
- 行业解决方案包（可打包交付）

---

## 🔄 整体特点

- **每阶段交付可运行软件**，非 PPT 或纯设计
- **复用现有系统**，不做推倒重来
- **AI 作为增强能力贯穿各阶段**，但不依赖其核心流程
- **架构渐进演进**：从集成门户 → 统一模型 → 开放平台
- **资源灵活投入**：各阶段可在有空闲人力时启动，结合项目需求落地
