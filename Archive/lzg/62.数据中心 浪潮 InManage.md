## 一、产品概览

**产品名称**：InManage（全称：Inspur Infrastructure Intelligent Management Platform）  
**开发商**：浪潮信息（Inspur Information）  
**定位**：面向超大规模数据中心的 **AIOps 智能运维平台**  
**核心使命**：实现 **IT基础设施的统一纳管、智能预警、根因分析与能效优化**  
**行业认证**：
- 唯一入选 **Gartner《中国AIOps市场指南》** 的服务器厂商自研软件（2023–2025）
- 通过 **等保三级、信创适配认证**（兼容麒麟、统信UOS、鲲鹏、昇腾等）

## 二、InManage 监控对象与数据维度

### ✅ 主要监控目标（聚焦 IT 层）

| 类别 | 具体设备/组件 | 监控数据示例 |
|------|----------------|-------------|
| **服务器** | 浪潮 NF/i48 系列等 x86 服务器 | CPU 使用率/温度、内存故障、硬盘 SMART/RAID 状态、电源功率、风扇转速、BMC 日志、固件版本 |
| **存储系统** | AS/HF 系列存储 | 存储池容量/IOPS/延迟、磁盘健康、控制器负载、RAID 重建进度 |
| **虚拟化层** | VMware、KVM、OpenStack | 虚拟机资源使用、宿主机负载、迁移事件 |
| **操作系统** | Linux / Windows（需代理） | 进程、磁盘 I/O、网络连接、服务状态 |
| **网络设备** | 交换机/路由器（有限支持） | 端口状态、流量、错误包（通过 SNMP） |
| **动环数据（新版本扩展）** | 空调、PDU、温湿度传感器 | 机柜进风温度、IT设备功耗、冷通道温度（需对接第三方动环系统） |

> 🔍 **数据采集方式**：  
> - 带外：IPMI、Redfish、SNMP  
> - 带内：轻量级 Agent（可选）  
> - API 对接：VMware vCenter、OpenStack、Prometheus 等

## 三、用户界面与核心功能

### 🖥️ 用户界面特点
- **Web-based 图形化控制台**，支持多终端访问
- **模块化仪表盘**：自定义展示资源健康度、告警热力图、能效趋势
- **3D 机房拓扑视图**（可选）：按机楼→机房→机柜→设备层级下钻
- **设备详情页**：硬件拓扑图 + 实时性能曲线 + 告警日志 + 远程控制入口

### ⚙️ 核心用户功能

| 功能类别 | 具体能力 |
|--------|--------|
| **统一监控** | 10万+节点、300万+指标集中可视化 |
| **智能告警** | 自研 ETF 无阈值算法，告警准确率 **95.26%**，支持多级通知（邮件/短信/Webhook） |
| **预测性维护** | 硬盘故障提前 **15天预警**（准确率 >98%），内存故障预测提升30% |
| **远程运维** | Web KVM、虚拟媒体挂载、远程开关机、BIOS 配置 |
| **批量操作** | 固件升级、BIOS 批量配置、OS 部署（集成 InCloud Deploy） |
| **能效管理** | IT 设备功耗统计、PUE 趋势分析、与动环联动优化制冷 |
| **自动化运维** | 剧本引擎（Playbook）支持自动巡检、故障自愈、容量预警 |
| **开放集成** | 提供 RESTful API，支持对接 Zabbix、ServiceNow、CMDB、DCIM 等 |

## 四、与主流平台对比：InManage vs EcoStruxure IT

| 维度 | **浪潮 InManage** | **施耐德 EcoStruxure IT** |
|------|------------------|--------------------------|
| **核心领域** | **IT基础设施智能运维（AIOps）** | **数据中心物理基础设施管理（DCIM）** |
| **监控重点** | 服务器、存储、虚拟化、OS 性能 | UPS、PDU、空调、温湿度、机柜空间、电力拓扑 |
| **数据粒度** | 部件级（如单条内存、硬盘） | 系统级（如整机柜功耗、冷通道温度） |
| **AI 能力** | 强：根因分析、无阈值告警、故障预测 | 中：预测性维护（基于设备寿命模型） |
| **国产化支持** | ✅ 完整信创生态适配 | ❌ 依赖国外 OS/硬件，本地化部署有限 |
| **典型客户** | 工行、阿里、腾讯、国家超算 | 苹果、微软、大型IDC运营商 |
| **部署模式** | 本地化为主，支持容器化 | 本地 + 云（EcoStruxure IT Expert） |
| **价格模型** | 按管理节点数计费，性价比高 | License + 订阅制，成本较高 |

> 💡 **关键区别**：  
> - **InManage = “管算力”** → 关注 **业务是否跑得稳、快、省**  
> - **EcoStruxure IT = “管电力与环境”** → 关注 **机房是否不断电、不高温、不漏水**

## 五、在数据中心监控体系中的定位

### 🏗️ 数据中心监控分层模型

```
┌──────────────────────┐
│   业务应用监控       │ ← APM（如 Bonree ONE）
├──────────────────────┤
│   IT基础设施监控     │ ← **InManage（核心）**
│   （服务器/存储/网络）│
├──────────────────────┤
│   物理基础设施监控   │ ← EcoStruxure IT / 国产DCIM
│   （电力/制冷/环境） │
└──────────────────────┘
```

### 🤝 实际部署模式（典型客户）

| 客户类型 | IT监控平台 | 设施监控平台 | 集成方式 |
|--------|-----------|-------------|--------|
| 金融数据中心 | **InManage** | 施耐德 EcoStruxure 或 安正信 DCIM | 通过 API 同步告警，共用大屏 |
| 互联网云厂商 | InManage + 自研可观测平台 | Vertiv Trellis 或 自研DCIM | 数据湖统一存储，AI跨层分析 |
| 政务云 | InManage（信创版） | 云智慧 DCIM | 工单系统联动，未深度集成 |

> ✅ **趋势**：头部客户正推动 **IT + 设施数据融合**，构建“从芯片到电网”的数字孪生闭环。

## 六、适用场景与选型建议

### ✅ 推荐使用 InManage 的场景：
- 需要 **国产化替代**（信创要求）
- 运营 **AI大模型训练集群** 或 **超大规模云平台**
- 对 **服务器故障预测、MTTR 缩短** 有强需求
- 希望 **降低PUE、实现绿色数据中心**

### ❌ 不推荐单独使用的场景：
- 纯边缘微型机房（无服务器集群）
- 仅需监控电力/空调（应选 DCIM）
- 无 IT 运维团队（InManage 需一定技术门槛）

### 🔧 最佳实践组合：
- **金融/政务客户**：InManage + 国产DCIM（如云智慧、安正信）
- **云服务商**：InManage + 自研可观测平台 + 轻量DCIM
- **追求一体化**：关注 **浪潮“元脑 InManage”** 新版本（已打通 IT 与动环）

## 八、总结

> **InManage 不是传统监控工具，而是面向 AI 时代的数据中心“智能运维大脑”**。  
> 它以 **服务器硬件深度感知 + AIOps 算法 + 信创合规** 为核心优势，在 **IT 层监控** 领域已成国内标杆。  
> 但在完整数据中心运维体系中，**需与 DCIM（如 EcoStruxure IT）协同使用**，才能实现从“算力”到“电力”的全栈智能管理。
