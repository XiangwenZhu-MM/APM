## 二、Bonree ONE 监控对象与数据维度

> ⚠️ 注意：**Bonree ONE 的监控重心不在服务器硬件或机房环境，而在“业务系统”本身**。

### ✅ 主要监控目标（聚焦 应用层 + 用户体验）

| 类别 | 监控对象 | 数据示例 |
|------|--------|--------|
| **前端用户体验** | Web / 移动 App 用户 | 页面加载时间、JS错误率、点击热力图、地域/运营商性能分布 |
| **后端服务** | 微服务、API、中间件 | 响应时间、吞吐量、错误率、依赖拓扑 |
| **调用链追踪** | 分布式事务 | 全链路调用路径、各节点耗时、数据库/缓存调用详情 |
| **基础设施指标** | 主机、容器、K8s | CPU/内存/磁盘（仅作为上下文，非核心） |
| **日志数据** | 应用日志、系统日志 | 错误堆栈、业务流水、自定义字段 |
| **业务指标** | 自定义业务 KPI | 订单成功率、支付转化率、登录失败次数 |

> 🔍 **数据采集方式**：
> - **无侵入探针**（Java/.NET/Go/Python 等主流语言自动注入）
> - **RUM SDK**（Web/移动端）
> - **Log Agent**（Filebeat/Fluentd 兼容）
> - **OpenTelemetry 原生支持**

## 三、用户界面与核心功能

### 🖥️ 用户界面特点
- **统一控制台**：四大信号（Logs/Metrics/Traces/RUM）在一个平台内无缝切换
- **全链路拓扑图**：自动绘制服务依赖关系，点击任一节点下钻至代码级性能瓶颈
- **智能告警中心**：基于动态基线（非静态阈值）触发告警，减少噪音
- **自然语言搜索**（NLS）：支持“过去1小时支付失败最多的接口？”等语义查询
- **自定义仪表盘**：拖拽式构建业务健康度看板（如“大促期间订单履约监控”）

### ⚙️ 核心用户功能

| 功能类别 | 具体能力 |
|--------|--------|
| **全链路追踪** | 自动识别微服务调用关系，定位慢请求根因（精确到 SQL 语句或方法） |
| **智能基线告警** | 基于历史行为自动学习正常模式，异常波动自动告警 |
| **日志智能分析** | 支持日志聚类、异常模式识别、关联 Trace ID |
| **用户体验监控**（RUM） | 实时监测真实用户访问性能，按地域/设备/网络细分 |
| **业务可观测性** | 将技术指标与业务 KPI 关联（如“API 延迟 >2s → 订单流失率上升”） |
| **根因推荐** | AI 引擎自动给出可能原因（如“数据库连接池耗尽”） |
| **开放集成** | 支持对接 Prometheus、Zabbix、ServiceNow、钉钉/企微告警 |

## 四、与 InManage、EcoStruxure IT 的对比

| 维度 | **Bonree ONE** | **浪潮 InManage** | **施耐德 EcoStruxure IT** |
|------|----------------|------------------|--------------------------|
| **监控层级** | **应用层 + 用户体验**（L7） | **IT基础设施层**（L1–L4：服务器/存储） | **物理设施层**（电力/制冷/空间） |
| **核心价值** | 保障 **业务连续性 & 用户满意度** | 保障 **服务器高可用 & 算力效率** | 保障 **机房物理安全 & 能效合规** |
| **数据来源** | 应用探针、RUM SDK、日志 | BMC/IPMI、OS Agent、Redfish | 智能PDU、温湿度传感器、UPS |
| **AI 能力** | 智能基线、根因推荐、日志聚类 | 故障预测（硬盘/内存）、无阈值告警 | 设备寿命预测、能效优化建议 |
| **典型用户** | SRE、DevOps、产品经理 | IT运维工程师、设施管理员 | 设施经理、电气工程师 |
| **是否管硬件？** | ❌ 不监控服务器硬件状态 | ✅ 深度监控CPU/内存/硬盘 | ❌ 不涉及IT设备 |
| **是否管环境？** | ❌ | ❌（新版本部分支持） | ✅ 核心能力 |

> 💡 **一句话总结三者分工**：
> - **EcoStruxure IT**：确保“机房不断电、不高温”  
> - **InManage**：确保“服务器不宕机、算得快”  
> - **Bonree ONE**：确保“用户能下单、支付不卡顿”

## 五、在数据中心/IT 运维体系中的定位

### 🏗️ 可观测性在现代运维架构中的位置

```
┌──────────────────────┐
│   用户体验监控 (RUM) │ ← Bonree ONE
├──────────────────────┤
│   应用性能监控 (APM) │ ← Bonree ONE（核心）
├──────────────────────┤
│   IT基础设施监控     │ ← InManage
├──────────────────────┤
│   物理基础设施监控   │ ← EcoStruxure IT
└──────────────────────┘
```

> ✅ **Bonree ONE 是“业务视角”的监控入口**，而 InManage 是“资源视角”的监控底座。二者常**协同使用**：
> - 当 Bonree ONE 发现“支付接口变慢”，可联动 InManage 查看底层服务器是否 CPU 打满或硬盘故障。
> - 反之，InManage 预警“某服务器硬盘将坏”，Bonree ONE 可评估该服务器承载的业务影响范围。

## 六、适用场景与选型建议

### ✅ 推荐使用 Bonree ONE 的场景：
- 采用 **微服务、云原生、Serverless** 架构
- 业务高度依赖 **线上用户体验**（如电商、金融App、SaaS）
- 需要 **快速定位分布式系统故障**（MTTR < 15分钟）
- 希望将 **技术指标与业务结果关联**（如“性能下降 → GMV 损失”）

### ❌ 不适合的场景：
- 纯传统单体应用（无分布式调用）
- 仅需监控服务器硬件状态（应选 InManage）
- 无前端用户交互的后台批处理系统

### 🔧 最佳实践组合：
- **互联网公司**：Bonree ONE + Kubernetes 监控 + InManage（混合云场景）
- **银行核心系统**：Bonree ONE（手机银行/网银） + InManage（信创服务器） + DCIM（机房）
- **DevOps 团队**：Bonree ONE 作为 CI/CD 流水线的质量门禁（性能回归检测）
